# Clash 订阅管理器产品需求文档

## 一、项目目标

本项目拟开发一套个人使用的 Clash 订阅管理器，用于对多个 Clash 订阅源进行统一管理、合并处理、脚本化加工与配置输出。系统面向个人多订阅、多设备、多节点管理场景，提供从“订阅导入”到“规则加工”再到“最终配置生成”的完整能力。

项目核心目标如下：

1. 统一管理多个 Clash 订阅
支持添加、编辑、启用、禁用和刷新多个 Clash 订阅源，便于集中维护多个机场订阅、自建代理订阅及本地导入配置。

2. 实现多个订阅的合并处理
支持将多个 Clash 订阅进行统一合并，生成一份新的配置文件，并对节点进行去重、重命名、过滤和整理，解决多个订阅分散使用的问题。

3. 通过 JavaScript 脚本控制订阅生成逻辑
支持使用 JavaScript 编写订阅加工脚本，对导入后的 Clash 配置进行二次处理，实现节点筛选、节点改名、节点分组、代理组生成等自定义逻辑，使配置生成过程具备高度可编程性和可扩展性。

4. 支持节点组合、分组与过滤
可根据地区、关键词、协议类型、来源订阅等维度，对节点进行自动分类和组合，快速生成如“香港节点”“日本节点”“新加坡节点”“自动选择”“全部节点”等常用代理组。

5. 自动生成最终 Clash YAML 配置
系统根据订阅源与 JS 处理脚本自动输出最终可用的 Clash YAML 配置文件，支持本地文件导出，也支持生成可供多设备使用的订阅链接。

6. 兼容多种订阅来源
除标准 Clash 在线订阅外，同时支持本地 YAML 文件导入，以及自建代理节点配置导入，便于统一接入不同来源的代理数据。

总体上，本项目的目标是：用 JavaScript 代码驱动 Clash 订阅的生成逻辑，实现多个订阅的灵活整合、自动加工和统一输出。

其基本处理模式可概括为：

`订阅A + 订阅B + 订阅C -> JS脚本处理 -> 生成新的 Clash 配置`

## 二、使用场景（核心）

本系统的典型使用流程围绕“导入订阅、加工节点、生成配置、分发订阅”展开，核心使用场景如下：

1. 场景一：管理多个机场订阅
用户通常会同时持有多个机场订阅地址，例如：

- 机场 A：`https://example-a.com/clash.yaml`
- 机场 B：`https://example-b.com/clash.yaml`
- 机场 C：`https://example-c.com/clash.yaml`

系统需要支持对这些订阅进行统一管理，用户可以执行以下操作：

- 添加新的订阅源
- 编辑订阅名称、地址和备注
- 启用或禁用某个订阅
- 手动刷新订阅内容
- 查看订阅对应的节点数量及更新时间

通过该场景，用户可以将原本分散的多个 Clash 订阅集中到一个系统中统一维护。

2. 场景二：合并多个订阅
当用户拥有多个机场或多个代理来源时，往往希望将多个订阅合并为一个统一配置。例如：

- 订阅 A
- 订阅 B
- 订阅 C

系统应支持将多个订阅解析后合并为一个新的 Clash 配置，并自动完成以下处理：

- 汇总全部节点
- 对重复节点进行去重
- 对节点名称进行规范化和重命名
- 按来源或地区进行整理分类
- 生成统一的代理组结构

通过该场景，用户无需分别维护多个订阅文件，只需使用一份合并后的最终配置即可。

3. 场景三：通过 JavaScript 控制节点加工逻辑
系统的核心特性之一，是支持用户通过 JavaScript 脚本定义订阅处理规则。例如：

```javascript
function transform(config) {
  config.proxies = config.proxies.filter(p =>
    p.name.includes("HK") ||
    p.name.includes("SG")
  )
  return config
}
```

在该场景下，用户可以通过脚本实现：

- 仅保留特定地区节点，如 HK、SG、JP、US
- 删除低质量或不需要的节点
- 批量修改节点名称
- 根据来源订阅添加统一前缀
- 动态生成代理组
- 调整最终 Clash 配置结构

该能力使用户不再依赖固定规则，而是可以通过代码完全控制 Clash 配置的生成过程。

4. 场景四：自动生成节点分组
用户通常希望系统自动生成适合 Clash 客户端使用的代理组，例如：

- 节点选择
- 自动选择
- 香港节点
- 日本节点
- 新加坡节点
- 美国节点
- 全部节点

系统应能基于节点名称、地区关键词、来源标签或用户脚本自动生成这些分组，并将对应节点填充到组内。这样可以显著减少手动维护 YAML 配置的工作量，并提升配置的可用性和一致性。

5. 场景五：生成最终 Clash 配置并提供订阅输出
在用户完成订阅导入、合并与脚本处理后，系统最终输出一份完整的 Clash YAML 配置文件，例如：

- `merged.yaml`

该输出结果应支持两种主要使用方式：

- 本地下载或导出 YAML 文件，供 Clash 客户端手动导入
- 生成新的订阅链接，供个人多设备统一订阅使用

这样，用户可以把系统作为自己的“订阅中台”，所有设备只订阅系统生成的统一链接，而不再直接依赖多个原始机场订阅。

## 三、功能需求

本系统定位为个人使用的 Clash 订阅管理与生成工具，功能上围绕“订阅管理、配置加工、结果生成、订阅分发”四个方面展开。

1. 订阅源管理
系统应支持多种订阅来源的统一接入与维护，具体包括：

- 支持添加在线 Clash 订阅地址
- 支持导入本地 Clash YAML 文件
- 支持导入自建代理节点配置
- 支持为每个订阅设置名称、备注、标签
- 支持启用或禁用某个订阅
- 支持手动刷新订阅内容
- 支持查看订阅更新时间、节点数量、状态信息

2. 订阅解析与合并
系统应能对多个订阅内容进行解析并统一合并，形成标准化的中间配置对象，具体包括：

- 解析多个 Clash YAML 订阅
- 提取 `proxies`、`proxy-groups`、`rules` 等结构
- 将多个订阅中的节点合并为统一节点列表
- 支持节点去重
- 支持节点重命名
- 支持保留来源信息，便于后续脚本区分
- 支持对异常订阅进行容错处理

3. JavaScript 脚本加工能力
系统的核心功能是允许用户通过 JavaScript 对配置生成过程进行控制，具体包括：

- 支持编写 `transform(config)` 风格的处理函数
- 支持对节点列表进行过滤、排序、改名、分组
- 支持动态生成 `proxy-groups`
- 支持修改 Clash 配置的基础字段
- 支持在脚本中访问订阅来源信息
- 支持脚本执行错误提示与日志输出

建议系统内部统一约定处理入口，例如：

```javascript
function transform(config, context) {
  return config
}
```

其中：

- `config` 表示当前合并后的 Clash 配置对象
- `context` 表示订阅来源、时间、用户参数等上下文信息

4. 节点分组与组合
系统应支持自动或脚本化生成常用节点分组，具体包括：

- 按地区自动分组，如香港、日本、新加坡、美国
- 按来源分组，如机场 A、机场 B、自建节点
- 按关键词分组
- 生成“全部节点”分组
- 生成“自动选择”分组
- 生成“手动选择”分组
- 支持组合多个分组形成最终代理组结构

5. 最终配置生成
系统应支持将处理后的配置输出为完整、可直接使用的 Clash YAML 文件，具体包括：

- 输出标准 Clash YAML
- 支持保存为本地文件
- 支持命名输出文件，如 `merged.yaml`
- 支持生成统一订阅链接
- 支持多设备重复拉取同一个生成结果

6. 基础辅助能力
为了便于长期使用，系统还应具备一定的辅助功能：

- 操作日志记录
- 配置版本管理或历史快照
- 订阅刷新失败提示
- 脚本调试与预览
- 一键重新生成配置

## 四、系统设计

从结构上看，本系统可以拆分为四个核心模块：订阅输入模块、配置处理模块、脚本执行模块、结果输出模块。

1. 订阅输入模块
负责接收并管理各类订阅源，统一拉取和解析配置内容。其职责包括：

- 管理在线订阅地址
- 管理本地 YAML 导入
- 管理自建节点导入
- 拉取远程订阅并缓存原始内容
- 将不同来源的数据解析为统一内部结构

2. 配置处理模块
负责把多个来源的节点和配置合并成一份中间数据，并完成基础清洗。其职责包括：

- 节点汇总
- 节点去重
- 节点重命名
- 基础字段规范化
- 来源标记补充

可将内部标准结构设计为：

```js
{
  proxies: [],
  proxyGroups: [],
  rules: [],
  metadata: {}
}
```

3. 脚本执行模块
这是系统的核心控制层，用于执行用户定义的 JavaScript 规则。其职责包括：

- 加载用户脚本
- 注入标准 `config` 对象
- 注入执行上下文 `context`
- 执行 `transform` 函数
- 捕获运行错误并输出日志
- 返回加工后的最终配置对象

4. 结果输出模块
负责把内部对象序列化成 Clash YAML，并通过文件或 HTTP 方式输出。其职责包括：

- 生成最终 YAML
- 输出本地文件
- 暴露订阅接口
- 提供最新配置内容给设备拉取

系统整体处理链路如下：

`订阅输入 -> 解析标准化 -> 合并清洗 -> JS脚本加工 -> YAML生成 -> 文件/订阅链接输出`

## 五、技术方案

考虑到项目目标明确、以个人使用为主，同时需要较强的可编程能力，建议采用 Node.js + JavaScript 作为主要技术栈。

1. 后端核心
建议使用 Node.js 构建主程序，原因如下：

- 原生适合执行 JavaScript 脚本
- 处理 YAML、HTTP 请求和文件操作都较方便
- 易于扩展为本地服务或轻量 Web 服务
- 与用户自定义脚本语言保持一致

可选技术：

- `Express` 或 `Fastify`：提供本地 HTTP 服务
- `axios`：拉取远程订阅
- `yaml`：解析和生成 YAML
- `vm` 或 `isolated-vm`：执行用户脚本
- `fs`：本地文件读写

2. 配置处理模型
建议系统内部采用“先解析成 JS 对象，再统一加工，最后输出 YAML”的方式，而不是直接拼接 YAML 文本。

处理流程建议为：

1. 拉取订阅内容
2. YAML 解析为对象
3. 提取与规范化节点数据
4. 合并多个配置
5. 调用用户脚本处理
6. 生成最终 YAML

这种方式更利于扩展，也更适合做去重、分组、过滤等逻辑。

3. 脚本执行安全设计
虽然项目为个人使用，但仍建议对脚本执行做边界控制，至少避免脚本直接影响系统主进程。建议方案如下：

- 使用沙箱执行脚本
- 仅暴露 `config` 与受控 `context`
- 不默认开放文件系统与网络访问
- 对脚本执行设置超时时间
- 对异常进行捕获并返回错误日志

4. 订阅输出方式
建议同时支持两种输出：

- 文件输出模式：生成本地 `merged.yaml`
- HTTP 订阅模式：提供如 `http://127.0.0.1:3000/sub/merged.yaml` 的访问地址

其中 HTTP 模式更适合个人多设备统一订阅，是长期使用时最实用的方案。

5. 推荐目录结构
可参考如下结构：

```text
proxy-manager/
  data/
    sources.json
    scripts/
    output/
    cache/
  src/
    server.js
    core/
      fetch.js
      parse.js
      merge.js
      transform.js
      generate.js
    routes/
    utils/
  package.json
```

目录含义如下：

- `data/sources.json`：保存订阅源列表
- `data/scripts/`：保存用户 JS 脚本
- `data/output/`：保存生成后的 YAML
- `data/cache/`：缓存原始订阅内容
- `src/core/`：核心处理逻辑

## 六、产品原型结构

从使用体验上，本系统可以先按“轻量后台工具”设计，优先保证功能闭环，再决定是否补充完整前端界面。

1. 方案一：命令行 + 本地服务
这是最适合第一阶段落地的形式。用户通过配置文件或命令操作系统功能，例如：

- 添加订阅
- 刷新订阅
- 执行脚本
- 生成 YAML
- 启动本地订阅服务

典型命令示例：

```bash
proxy-manager add-source
proxy-manager refresh
proxy-manager build
proxy-manager serve
```

优点：

- 实现快
- 结构简单
- 便于调试
- 符合个人工具属性

2. 方案二：简易 Web 管理界面
在命令行方案基础上，可以增加一个本地 Web 面板，提升可用性。页面可以包括以下几个部分：

- 订阅管理页
- 脚本编辑页
- 配置预览页
- 输出管理页

3. 核心页面结构建议
如果后续做 Web UI，可按如下结构规划：

```text
首页 / 仪表盘
订阅管理
脚本管理
构建记录
输出配置
系统设置
```

4. 核心数据对象设计
可初步定义以下数据模型：

订阅源 Source：

```js
{
  id: "src_001",
  name: "机场A",
  type: "remote",
  url: "https://example.com/clash.yaml",
  enabled: true,
  tags: ["airport"],
  updatedAt: ""
}
```

脚本 Script：

```js
{
  id: "script_default",
  name: "默认处理脚本",
  content: "function transform(config, context) { return config }",
  updatedAt: ""
}
```

构建结果 Build：

```js
{
  id: "build_001",
  sourceIds: ["src_001", "src_002"],
  outputFile: "merged.yaml",
  proxyCount: 120,
  groupCount: 8,
  createdAt: ""
}
```

## 七、非功能需求

除核心功能外，系统还应满足一定的非功能性要求，以保证其在个人长期使用中的稳定性、可维护性和可扩展性。

1. 易用性
系统应尽量降低使用门槛，使用户能够以较低成本完成订阅导入、脚本编写、配置生成和结果分发。即使用户不具备复杂开发背景，也应能通过少量配置完成基础使用。

2. 稳定性
系统在面对订阅拉取失败、YAML 格式异常、脚本执行报错等情况时，应具备较好的容错能力，不能因单一订阅或单次执行异常导致整个系统不可用。

3. 可扩展性
系统设计应采用模块化结构，便于后续扩展更多订阅来源、脚本能力、节点处理规则、输出格式和前端管理界面。

4. 可维护性
核心逻辑应分层清晰，订阅管理、解析合并、脚本执行、配置输出等模块职责明确，便于后期排查问题和持续迭代。

5. 可观测性
系统应记录关键执行过程中的日志信息，例如订阅拉取结果、解析异常、脚本执行错误、构建结果摘要等，便于调试和定位问题。

6. 性能要求
由于系统主要面向个人使用，其性能目标不需要极端高并发，但应保证在常见场景下具备良好响应能力。例如：

- 支持同时处理多个订阅源
- 支持百级到千级节点规模
- 单次构建应在可接受时间内完成
- 刷新与生成过程不应明显阻塞使用

7. 安全性
尽管系统主要为个人使用，但由于涉及在线订阅地址、脚本执行和配置生成，仍需考虑基础安全控制：

- 用户脚本应在受控环境中执行
- 不默认开放任意系统命令能力
- 不泄露原始订阅地址和敏感凭证
- 生成接口应具备最基本的访问控制或路径隔离能力

## 八、数据库 / 文件存储设计

考虑到本项目定位为个人工具，初期不一定需要引入完整数据库，采用“文件存储为主、数据库可选”的方案更为合适。

1. 推荐存储策略
建议优先采用本地文件存储，保存核心配置和运行结果，包括：

- 订阅源定义
- 用户脚本
- 原始订阅缓存
- 生成后的 YAML 文件
- 构建日志和历史记录

这种方式部署简单、迁移方便，适合单用户长期维护。

2. 建议目录结构
可采用如下数据目录结构：

```text
data/
  sources.json
  scripts/
    default.js
  cache/
    src_001.yaml
    src_002.yaml
  output/
    merged.yaml
  builds/
    build-20260314-001.json
  logs/
    app.log
```

3. 核心存储内容说明

`sources.json`
用于保存订阅源列表及其基本信息，例如：

```json
[
  {
    "id": "src_001",
    "name": "机场A",
    "type": "remote",
    "url": "https://example.com/clash.yaml",
    "enabled": true,
    "tags": ["airport"],
    "updatedAt": "2026-03-14T10:00:00Z"
  }
]
```

`scripts/default.js`
用于保存用户编写的默认转换脚本，例如：

```javascript
function transform(config, context) {
  return config
}
module.exports = { transform }
```

`cache/*.yaml`
用于缓存订阅原始内容，便于：

- 提升重复构建效率
- 在远程订阅异常时使用上一次缓存
- 做构建对比和问题排查

`output/merged.yaml`
用于保存最终生成的 Clash 配置文件，作为系统的核心输出结果。

`builds/*.json`
用于记录每次构建的元数据，例如：

```json
{
  "id": "build_001",
  "createdAt": "2026-03-14T10:10:00Z",
  "sourceIds": ["src_001", "src_002"],
  "proxyCount": 128,
  "groupCount": 6,
  "outputFile": "merged.yaml",
  "status": "success"
}
```

4. 数据库扩展建议
如果后续增加更多能力，例如：

- 多脚本方案切换
- 多输出配置方案
- 更细的构建历史查询
- Web 界面复杂筛选

则可以再引入轻量数据库，例如：

- `SQLite`：适合单机本地持久化
- `LowDB`：适合轻量 JSON 数据管理

初期建议仍以文件存储为主，降低复杂度。

## 九、接口设计

如果系统支持本地 HTTP 服务，则建议设计一组简单明确的接口，用于订阅管理、构建执行和结果输出。

1. 订阅源管理接口

获取订阅源列表：

```http
GET /api/sources
```

返回示例：

```json
[
  {
    "id": "src_001",
    "name": "机场A",
    "type": "remote",
    "enabled": true
  }
]
```

新增订阅源：

```http
POST /api/sources
```

请求示例：

```json
{
  "name": "机场A",
  "type": "remote",
  "url": "https://example.com/clash.yaml",
  "enabled": true
}
```

更新订阅源：

```http
PUT /api/sources/:id
```

删除订阅源：

```http
DELETE /api/sources/:id
```

刷新指定订阅源：

```http
POST /api/sources/:id/refresh
```

2. 脚本管理接口

获取当前脚本：

```http
GET /api/scripts/current
```

保存脚本：

```http
PUT /api/scripts/current
```

请求示例：

```json
{
  "content": "function transform(config, context) { return config }"
}
```

测试执行脚本：

```http
POST /api/scripts/test
```

用于对当前合并配置执行预处理，返回执行结果或错误信息。

3. 构建接口

执行构建：

```http
POST /api/build
```

功能说明：

- 拉取启用的订阅
- 合并配置
- 执行脚本
- 输出最终 YAML

返回示例：

```json
{
  "success": true,
  "buildId": "build_001",
  "proxyCount": 128,
  "outputFile": "merged.yaml"
}
```

获取构建记录：

```http
GET /api/builds
```

获取构建详情：

```http
GET /api/builds/:id
```

4. 输出接口

获取最终 YAML：

```http
GET /sub/merged.yaml
```

该接口用于 Clash 客户端直接订阅，是系统最关键的对外输出接口。

下载最终 YAML：

```http
GET /api/output/download
```

获取输出信息：

```http
GET /api/output/info
```

返回内容可以包括：

- 文件名
- 最后生成时间
- 节点数量
- 分组数量
- 可订阅地址

5. 系统接口

获取系统状态：

```http
GET /api/system/status
```

用于查看：

- 服务运行状态
- 最近一次构建结果
- 订阅刷新时间
- 日志摘要

## 十、最小可行版本 MVP 清单

为了尽快落地项目，建议先实现一个最小可用版本，确保从“输入订阅”到“输出 YAML”的核心链路可以跑通。

1. MVP 必做功能

- 支持添加多个远程 Clash 订阅
- 支持启用和禁用订阅
- 支持手动刷新订阅
- 支持读取本地 YAML 文件
- 支持合并多个订阅中的 `proxies`
- 支持节点基础去重
- 支持加载一个 JS 脚本执行 `transform(config, context)`
- 支持生成最终 `merged.yaml`
- 支持通过本地 HTTP 地址输出订阅内容

2. MVP 建议简化点

为了加快开发，第一版可以暂时不做：

- 完整前端页面
- 复杂权限系统
- 自动定时刷新
- 构建历史可视化
- 多用户支持
- 高级脚本调试器
- 完整规则集管理

3. MVP 的最小使用流程

1. 用户添加多个订阅源
2. 系统拉取并缓存 YAML
3. 系统合并所有启用订阅的节点
4. 系统执行用户自定义 JS 脚本
5. 系统生成 `merged.yaml`
6. 用户通过本地文件或订阅链接在 Clash 客户端中使用

4. MVP 验收标准
最小版本完成后，应至少满足以下验收要求：

- 能成功管理 3 个以上订阅源
- 能正确合并多个订阅中的节点
- 能通过 JS 脚本过滤指定节点
- 能自动输出一份合法的 Clash YAML
- 能通过本地订阅链接供客户端拉取
- 在单个订阅异常时仍能完成整体构建或给出明确错误

## 十一、后续演进方向

在 MVP 跑通后，系统可以进一步扩展以下能力：

1. 支持自动定时刷新与自动构建
2. 支持多个脚本方案和多个输出配置
3. 支持节点测速与自动排序
4. 支持规则集模板和地区分组模板
5. 支持 Web 可视化管理界面
6. 支持订阅访问令牌或路径签名
7. 支持生成不同设备专用配置
8. 支持更多代理配置格式的导入与转换
